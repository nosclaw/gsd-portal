## 1. 背景与问题定义

### 1.1 项目背景
Nosclaw 需要一个私有、安全、可审计的 AI 开发门户，用于统一管理团队成员对 GSD 工作空间的访问、身份、环境与产出归属。当前团队在使用 AI 开发环境时，存在访问入口分散、环境不一致、用户审批缺失、Git 提交归属不清、运维排障成本高的问题。

### 1.2 核心问题
1. 团队成员进入 GSD 的路径不统一，存在账号共享、人工分发地址或手动配置的问题。
2. 开发环境初始化依赖个人习惯，`standards`、依赖工具、偏好配置难以标准化。
3. AI 代操作后的 Git 提交归属不稳定，无法清晰追踪到实际发起人。
4. 缺少正式的准入与审批流程，平台无法控制谁可以访问系统、何时获得权限、何时被停用。
5. 缺少统一审计日志，管理员难以追踪用户登录、审批、工作空间启动、异常退出等关键行为。

### 1.3 问题陈述
GSD Portal 需要解决的不是“如何启动一个 GSD 进程”，而是“如何让组织以受控、可审计、低摩擦的方式，把正确的人带入正确的 AI 开发环境，并确保其操作与身份、工作区、权限边界一一对应”。

## 2. 目标用户与 Jobs To Be Done

### 2.1 目标用户
| 角色 | 说明 | 核心诉求 |
| --- | --- | --- |
| Platform Root Admin | 平台级运维/负责人 | 初始化系统、创建租户、处理高权限操作 |
| Tenant Admin | 租户管理员/团队负责人 | 审批成员、管理租户内用户、查看审计记录 |
| Developer Member | 研发成员 | 一键进入自己的 GSD 工作空间并开始开发 |
| Security / Ops | 安全或运维协作者 | 追踪风险事件、排查故障、回收权限 |

### 2.2 核心 Jobs To Be Done
- 作为租户管理员，我需要只让被批准的成员访问 GSD Portal，这样团队资源和代码环境不会被未授权用户使用。
- 作为开发成员，我需要登录后快速进入自己的工作空间，这样我不必手动配置端口、Token 和环境。
- 作为组织负责人，我需要 AI 产生的提交、操作和异常都能回溯到具体人员，这样我可以做审计与责任追踪。
- 作为平台运维，我需要统一环境初始化和回收机制，这样平台能在安全与成本之间保持平衡。

## 3. 产品目标与非目标

### 3.1 产品目标
1. 建立一个私有部署、身份感知、可审计的 GSD 访问入口。
2. 为每个用户提供独立工作空间，并确保跨用户、跨租户之间不存在未授权访问。
3. 让用户通过 Portal 单点进入 GSD，避免手动复制 Token 或访问临时端口。
4. 在 GSD 运行时注入用户身份，使 AI 代操作产物可准确归属到实际用户。
5. 通过统一环境初始化机制，保证团队工作空间具备一致的基础标准。
6. 提供官方支持的 Docker Compose 自托管部署方案，降低平台落地与运维门槛。
7. Portal 前端统一使用 HeroUI 组件体系，并以 `ui/design.png` 作为界面设计参考稿。

### 3.2 非目标
1. 本期不做面向公众的开放注册型 SaaS 平台。
2. 本期不做计费、套餐、资源售卖或用量结算能力。
3. 本期不做跨租户协作、跨租户代码共享、跨租户权限委派。
4. 本期不做企业级 SSO、SCIM、外部 IAM 深度集成，用户名密码体系为 MVP 默认方案。
5. 本期不做多集群调度与跨主机弹性编排，优先完成单部署、单控制面的产品闭环。
6. 本期不要求 Kubernetes 或 Helm 成为官方部署前提，Docker Compose 为默认推荐方案。

## 4. 产品范围与关键定义

### 4.1 关键定义
| 名词 | 定义 |
| --- | --- |
| Tenant | 一个组织或团队级隔离单元，拥有自己的成员、审批记录、工作空间与配置边界 |
| User | 平台登录主体，属于且仅属于一个 Tenant |
| Workspace | 为某个 User 独占分配的 GSD 运行环境与目录 |
| Portal Session | 用户登录 Portal 后获得的会话 |
| GSD Session | Portal 为用户桥接到 GSD 时产生的下游会话，包含访问令牌以及 GSD 提供的续期机制，例如 refresh token |

### 4.2 范围边界
1. v1 中每个用户仅允许拥有一个活跃工作空间。
2. v1 中每个用户仅允许归属于一个租户。
3. v1 中租户由 Platform Root Admin 创建，租户管理员由 Root Admin 指定或转交。
4. v1 默认以邀请制或管理员预创建用户为主；如需开放注册，必须由租户管理员显式开启。

### 4.3 隔离边界
平台必须对以下维度进行隔离：
- 身份与权限隔离
- 工作目录隔离
- GSD 会话隔离
- 反向代理路由隔离
- 审计日志检索权限隔离
- 开发应用端口访问隔离

## 5. 关键用户流程

### 5.1 平台初始化
1. 平台首次部署后，由运维通过环境变量、初始化脚本或 CLI 创建 Root Admin。
2. 不允许“首个注册成功用户自动成为管理员”。
3. Root Admin 创建首个 Tenant，并指定 Tenant Admin。

### 5.2 用户加入与审批
1. Tenant Admin 创建邀请或开启租户注册入口。
2. 用户提交注册信息，包括用户名、显示名、邮箱、密码。
3. 用户提交后状态进入 `PENDING`。
4. Tenant Admin 可执行 `APPROVE`、`REJECT`、`SUSPEND`、`UNSUSPEND`。
5. 仅 `APPROVED` 用户可登录并使用工作空间能力。
6. `REJECTED` 用户需看到拒绝结果与后续处理指引。
7. `SUSPENDED` 用户的现有 Portal Session 与 GSD Session 必须被立即失效。

### 5.3 工作空间启动与进入
1. `APPROVED` 用户登录 Portal 后，可点击“启动工作空间”。
2. 系统检查用户状态、租户状态、资源配额、是否已有运行中的工作空间。
3. 若用户已有运行中的工作空间，系统直接恢复会话并进入该空间。
4. 若用户没有运行中的工作空间，系统执行端口分配、环境准备、GSD 启动、健康检查、代理绑定。
5. 仅在 GSD 健康检查通过后，Portal 才可跳转用户进入 GSD 界面。
6. Portal 必须托管 GSD 的访问令牌与 refresh token 续期流程，在用户持续活跃且 refresh token 仍有效时，自动完成无感续期，不得因短时访问令牌过期而中断当前会话。
7. 若续期失败，系统应先尝试有限次自动重试；重试仍失败时，需向用户明确提示“会话失效，需要重新连接”，同时不得误报为工作空间已停止。
8. 启动失败时，用户应看到明确失败原因与可重试动作，系统不得遗留孤儿进程、脏 Token 或被占用端口。

### 5.4 工作空间停止、重启与回收
1. 用户可手动停止或重启自己的工作空间。
2. 管理员可对异常工作空间执行强制停止。
3. 系统应支持空闲回收策略，默认在连续空闲 60 分钟后自动停止工作空间。
4. 工作空间停止后，GSD Session 立即失效，运行端口释放。
5. 工作空间目录默认持久化保留，不因停止而删除。

### 5.5 身份注入与归属追踪
1. 启动工作空间时，系统必须注入用户级 Git 身份信息，包括 `GIT_AUTHOR_NAME`、`GIT_AUTHOR_EMAIL`、`GIT_COMMITTER_NAME`、`GIT_COMMITTER_EMAIL`。
2. Git 身份应来源于平台用户资料或租户策略，不得写死为系统账号。
3. 平台应保留“某次工作空间启动使用了哪个平台用户身份”的审计记录。
4. 如技术条件允许，应补充记录 AI 触发提交的 commit SHA 与关联用户。

## 6. 用户状态机与工作空间状态机

### 6.1 用户状态
| 状态 | 含义 | 允许行为 |
| --- | --- | --- |
| `INVITED` | 已被管理员邀请但未完成注册 | 可完成注册 |
| `PENDING` | 已注册，待管理员审批 | 不可进入工作空间 |
| `APPROVED` | 已通过审批 | 可正常登录与使用 |
| `REJECTED` | 已拒绝 | 不可使用，可查看结果 |
| `SUSPENDED` | 已停用 | 不可使用，强制失效现有会话 |

### 6.2 工作空间状态
| 状态 | 含义 | 说明 |
| --- | --- | --- |
| `STOPPED` | 未运行 | 默认状态 |
| `STARTING` | 启动中 | 正在分配端口、准备环境、拉起 GSD |
| `RUNNING` | 运行中 | 可通过 Portal 进入 |
| `STOPPING` | 停止中 | 正在清理 GSD 与代理 |
| `ERROR` | 启动或运行异常 | 需展示错误并允许重试 |
| `EXPIRED` | 因空闲回收或策略失效 | 需重新启动 |

## 8. 非功能需求

### 8.1 安全性
1. 平台必须避免跨租户、跨用户访问工作目录、会话、代理路径与日志数据。
2. Portal 不得把 GSD Session 明文写入前端日志、浏览器持久存储或可复用 URL。
3. 密码必须采用业界认可的安全哈希方案存储。
4. 所有高风险操作必须进入审计日志，包括审批、停用、强制停止工作空间、租户状态变更。
5. Docker Compose 部署时使用的敏感配置不得硬编码在镜像或 Compose 文件中，必须通过环境变量、独立 env 文件或外部密钥注入。

### 8.2 可靠性
1. 工作空间启动失败后，系统必须自动清理端口占用、临时 Token 与孤儿进程。
2. Portal 与工作空间状态必须最终一致，禁止前端显示“运行中”但后端实例已退出。
3. 宿主机重启后，系统应能识别并纠正失真的工作空间状态。
4. 在 refresh token 有效且下游 GSD 可用的前提下，活跃用户会话必须保持连续，不得因访问令牌自然过期而被迫重新登录。
5. Docker Compose 部署后，关键服务应具备自动重启策略；宿主机重启后，Portal 与数据库应可自动恢复。

### 8.3 性能
1. 已初始化工作空间的再次启动时间，目标为 p95 不高于 30 秒。
2. 首次冷启动并完成基础环境准备，目标为 p95 不高于 5 分钟。
3. 工作空间启动成功率目标不低于 99%。

### 8.4 可运营性
1. 管理员应可在后台快速定位“谁在什么时候做了什么操作”。
2. 平台应具备基本容量监控能力，至少覆盖注册用户数、活跃工作空间数、启动失败数、异常退出数。
3. 审计日志默认保留 90 天。
4. 部署文档必须覆盖初始化、升级、回滚、备份与恢复的基本操作步骤，确保非研发角色也能完成标准部署。

### 8.5 界面体验
1. Portal 前端不得混用多个 UI 组件体系，HeroUI 为默认且唯一的基础组件来源。
2. 页面实现应优先遵循 `ui/design.png` 的视觉层级、信息密度、卡片组织和主操作路径。
3. 登录页、工作空间控制台、管理员列表页在桌面端必须具备一致的导航、间距、状态标签和操作反馈风格。

## 9. 默认配置与产品约束

### 9.1 默认端口规划
以下为默认部署值，必须支持通过部署配置覆盖：

| 端口区间 | 用途 |
| --- | --- |
| `29000` | Portal Web UI |
| `30000 - 30999` | GSD 工作空间 Web 端口 |
| `50000 - 59999` | 用户开发应用端口 |

### 9.2 目录与配置约束
1. 工作空间根目录必须是可配置项，不在 PRD 中绑定固定系统用户路径。
2. 平台必须支持“共享团队标准 + 用户私有偏好”的双层配置模式。
3. 用户开发应用端口默认只对该用户本人可见；跨用户共享访问不在 MVP 范围内。

### 9.3 Docker Compose 部署约束
1. Docker Compose 为 MVP 官方支持的默认部署方式，部署形态为单机、自托管、单控制面。
2. Compose 编排至少覆盖 Portal Web、反向代理、持久化存储挂载以及运行工作空间调度所需的基础服务。
3. 必须提供持久化卷策略，至少覆盖数据库文件、工作空间目录、审计日志或运行日志。
4. 必须支持通过环境变量或独立 env 文件覆盖域名、端口、存储路径、管理员初始化参数及安全配置。
5. 不同环境的 Compose 项目名、网络、数据卷命名应可区分，避免 `local`、`staging`、`production` 之间相互污染。
6. 部署产物应支持版本化升级，不要求 Kubernetes，但必须支持可重复部署与失败后人工回滚。

## 10. 成功指标

| 指标 | 定义 | 目标 |
| --- | --- | --- |
| 审批完成时效 | 从用户提交注册到管理员完成审批的中位时长 | `<= 4` 个工作小时 |
| 激活率 | `APPROVED` 用户在 24 小时内成功启动至少一次工作空间的比例 | `>= 70%` |
| 工作空间启动成功率 | 启动请求最终进入 `RUNNING` 的比例 | `>= 99%` |
| 热启动时延 | 已初始化工作空间再次启动的 p95 时延 | `<= 30s` |
| 会话续期成功率 | 需要触发 refresh token 续期的 GSD 会话中，续期成功且用户无感知中断的比例 | `>= 99.9%` |
| Git 归属准确率 | AI 产生提交中，正确记录为当前用户的比例 | `100%` |
| 严重隔离事故数 | 跨用户或跨租户未授权访问事件 | `0` |
| Docker Compose 部署成功率 | 按官方文档执行标准部署后，可在目标环境正常启动并通过健康检查的比例 | `>= 95%` |

## 12. 风险与依赖

### 12.1 关键依赖
1. GSD 本身是否支持稳定获取并桥接下游会话、访问令牌以及 refresh token 续期能力。
2. `nosclaw/dev-env/setup.sh` 的执行时长、幂等性与失败恢复能力。
3. 单机资源是否足以承载目标并发工作空间数。
4. Docker Compose 部署场景下，宿主机对工作空间目录挂载、权限模型与网络策略的支持程度。

### 12.2 主要风险
1. 如果工作空间仅做目录隔离而缺少操作系统级隔离，可能无法满足“安全多租户”的承诺。
2. 如果 Portal 对 GSD Session 与 refresh token 的托管策略设计不当，可能产生横向越权、会话泄漏或异常续期风险。
3. 如果审批、停用和会话撤销链路不完整，平台将难以满足企业内部的准入管控要求。
4. 如果冷启动时间过长，用户将绕开 Portal，回到手动使用环境的旧路径。
5. 如果 Docker Compose 部署缺少健康检查、持久化和升级指引，私有部署用户将面临高运维门槛和较高的首次部署失败率。

## 13. 开放问题
1. v1 是否需要支持邮箱验证，还是仅依赖管理员审批作为准入控制。
2. 工作空间隔离最终采用 Linux 用户级隔离、容器级隔离，还是两者结合。
3. 是否需要在 v1 中支持租户管理员自定义 Git 邮箱域名策略。
4. 开发应用端口是否需要提供“分享给同租户成员”的受控访问能力。
5. 审计日志是否需要接入外部 SIEM 或告警系统。
6. GSD 的 refresh token 生命周期、轮转规则、吊销机制和错误码规范是否稳定且可被 Portal 正确消费。
7. 是否需要在 V1 中提供自动备份、自动回滚或部署锁，以增强 Docker Compose 升级安全性。

## 14. 附录：当前技术方向
以下为当前实现方向，用于研发评审，不作为产品需求本身的唯一约束：

- **Framework**：Next.js 16.2.1 (App Router)
- **Database**：SQLite3 + Drizzle ORM
- **Auth**：Auth.js (Credentials Provider)
- **Orchestrator**：Node.js Child Process + Reverse Proxy
- **Deployment**：Docker Compose (official self-hosted deployment mode for MVP)
- **UI**：HeroUI (`heroui-inc/heroui`) + Tailwind CSS

## 15. 评审结论标准
当以下问题都能被回答时，本 PRD 才可进入开发排期评审：
1. 谁可以创建租户、审批用户、停用用户，边界是否清晰。
2. 一个用户如何从注册走到可用，异常分支是否定义完整。
3. 一个工作空间从启动到回收的完整状态是否定义清楚。
4. “安全多租户”具体由哪些隔离边界保证，而不是停留在口号层面。
5. 上线后用什么指标判断 GSD Portal 是否真的被采用且运行稳定。
