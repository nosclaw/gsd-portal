# GSD Team Automated Development Best Practices

A comprehensive guide for fully automated parallel development using multiple human developers + multiple GSD AI Agent teams, driven by a single PRD document.

Applicable to all programming languages and project types.

---

## 1. Core Philosophy

```
Goal: Humans only run /gsd auto — AI Agent teams handle all development,
commits, collaboration, and merging fully autonomously.
```

| Principle | Description |
|-----------|-------------|
| **PRD is the only human input** | Tech selection, PREFERENCES.md, milestone breakdown — all auto-generated by GSD |
| **Git is the only collaboration channel** | All teamwork flows through commit → push → PR → review → merge |
| **Main branch is protected** | Merges only via PR, no direct push allowed |
| **Git Worktree for parallel isolation** | Each developer works in an isolated worktree, zero conflicts |
| **Dependencies first, then maximize parallelism** | Dependent modules complete first, independent modules run in parallel |
| **Tech Manager handles integration** | A dedicated GSD AI Agent owns PR review, conflict resolution, merging, and acceptance |

### Tech Manager (GSD AI Agent)

A dedicated GSD AI Agent that **does not write business code** — only handles integration and coordination:

| Responsibility | Actions |
|---------------|---------|
| **PR Review** | Review every developer's PR for code quality and standards compliance |
| **Conflict Resolution** | Resolve merge conflicts when multiple PRs overlap, preserving both sides' intent |
| **Merge to Main** | Merge approved PRs into the protected main branch |
| **Acceptance Verification** | Check against PRD.md acceptance criteria item by item |
| **Dependency Coordination** | Monitor dependency module completion, notify downstream developers to start |
| **CI Monitoring** | Check CI/CD status, fix integration issues |
| **Release Preparation** | Run final integration tests, prepare release after all PRs merge |

Tech Manager's PREFERENCES.md configuration:

```markdown
## Role
You are the Tech Manager, responsible for code integration and team coordination.
You do NOT write business code. Your responsibilities are:
PR review, conflict resolution, merging, and acceptance verification.

## always_use_skills
- review
- standards
- github

## custom_instructions
- Continuously monitor all open PRs
- Review code quality, security, and standards compliance
- When resolving merge conflicts, preserve intent from both sides
- Ensure CI passes before merging
- Verify against PRD.md acceptance criteria item by item
- Comment on PRs with required changes when issues are found
```

## 3. GSD Configuration Hierarchy

GSD uses a three-tier configuration system:

### Tier 1: Global (`~/.gsd/`) — System-wide defaults

```
~/.gsd/
├── preferences.md              # Global preferences (mode, models, default skills)
├── standards/                  # Engineering standards
├── agent/
│   ├── auth.json               # API keys
│   ├── settings.json           # Provider, model, thinking level
│   └── skills/                 # Installed skills (symlinked from skillshare)
└── projects/                   # Per-project state (auto-managed by GSD)
    └── {HASH_ID}/
        ├── gsd.db              # Project state database
        ├── PROJECT.md          # Project summary
        ├── STATE.md            # Current state
        ├── DECISIONS.md        # Technical decisions
        ├── KNOWLEDGE.md        # Project knowledge
        ├── preferences.md      # Project-level preferences
        ├── milestones/         # Milestones
        │   └── M001/
        │       ├── CONTEXT.md
        │       └── DECISIONS.md
        └── activity/           # Session logs
```

### Tier 2: Workspace (auto-created by Portal)

```
/home/{username}/.gsd/
├── web-preferences.json
└── agent/
    ├── auth.json
    └── settings.json
```

### Tier 3: Project (project root — highest priority)

```
project/
├── PRD.md                      # The only human input
├── PREFERENCES.md              # Auto-generated by GSD Phase 0
└── src/                        # Source code
```

### Configuration Priority

```
Project PREFERENCES.md  >  ~/.gsd/projects/{hash}/preferences.md  >  ~/.gsd/preferences.md
```

## 4. Complete Workflow

### Phase 0: Automated Planning (Single Agent)

The project lead provides PRD.md and developer count. GSD automates all planning:

```
Read PRD.md and execute Phase 0 initialization.

Number of developers: 3 (excluding Tech Manager)

1. Analyze requirements, output tech stack recommendation (language, framework,
   database, deployment) with reasoning
2. Based on the stack, auto-generate PREFERENCES.md (skills config, project constraints)
3. Split requirements into parallelizable independent modules for 3 developers:
   - Identify inter-module dependencies
   - Dependent modules → Phase 2a (complete first)
   - Independent modules → Phase 2b (all parallel after dependencies are done)
   - Each module assigned to one developer
4. Output task assignment table: Developer # → Module → Branch name → Task list
5. Initialize Git repository:
   - Set up main branch protection rules
   - Create feature branches for each developer

Constraints:
- All code must be real business logic — no mock/stub
- Follow engineering standards in ~/.gsd/standards/
- Number of parallel tasks = number of developers, maximize parallelism
```

**Phase 0 output example:**

```
=== Task Assignment Table ===

Dependency Layer (Phase 2a — must complete first):
  Developer 1 → Shared foundation
    Branch: feat/foundation
    Tasks: Project init, shared types, DB schema, base components

Parallel Layer (Phase 2b — all parallel after dependency layer merges):
  Developer 1 → User module
    Branch: feat/user-module
    Tasks: Registration/login, user management, permissions

  Developer 2 → Payment module
    Branch: feat/payment-module
    Tasks: Order creation, payment integration, billing

  Developer 3 → Admin panel
    Branch: feat/admin-module
    Tasks: Reports, system settings, audit log
```

### Phase 1: Skeleton + Dependency Layer (Single Agent)

One developer runs `/gsd auto` to complete the project skeleton and shared dependencies:

```
Read PRD.md and PREFERENCES.md.
You are Developer 1, responsible for Phase 1 + Phase 2a (dependency layer).

1. Initialize project (per tech stack selection)
2. Configure lint, type checking, git hooks, CI
3. Create shared type definitions and base components
4. Create database schema and migrations
5. Complete all code that other modules depend on

Workflow:
- Work on feat/foundation branch
- Commit + push after each task
- Create PR when all tasks are done
- Wait for Tech Manager to merge to main, then notify other developers to start
```

### Phase 2: Parallel Development (Multiple Agents, Git Worktree)

After the dependency layer merges to main, all developers **start simultaneously**:

**On each developer's machine:**

```bash
# Clone project (if not already done)
git clone <repo-url>
cd project

# Create Git Worktree for isolated working directory
git worktree add ../project-feat-user-module feat/user-module

# Enter worktree directory
cd ../project-feat-user-module

# Start GSD fully autonomous mode
gsd auto
```

**Each GSD Agent receives instructions (determined by PREFERENCES.md + milestones):**

```
Read PRD.md and PREFERENCES.md.
You are Developer 2, responsible for the payment module (feat/payment-module).

Task list:
1. Order creation API and pages
2. Payment gateway integration
3. Billing management

Workflow:
- Work on feat/payment-module branch
- Commit + push after each completed feature
- Run all related tests
- Create PR to main when all tasks are done
- PR will be reviewed and merged by Tech Manager
```

### Phase 3: Tech Manager Integration

The Tech Manager Agent runs continuously, handling all integration work:

```
You are the Tech Manager. Execute this work loop continuously:

1. Check all open PRs
2. For each PR:
   a. /review — code quality audit
   b. Check CI status
   c. Verify against PRD.md acceptance criteria item by item
   d. If issues found → comment on PR requesting changes
   e. If approved → merge to main
3. If merge produces conflicts:
   a. Analyze conflict cause
   b. Resolve conflict, preserving intent from both sides
   c. Commit merge result
4. After all modules are merged:
   a. Run full test suite
   b. Final verification against all PRD.md acceptance criteria
   c. Output acceptance report
```

## 5. Git Workflow

### Branch Strategy

```
main (protected)
  │
  ├── feat/foundation          ← Developer 1 (Phase 2a, dependency layer)
  │
  ├── feat/user-module         ← Developer 1 (Phase 2b, parallel)
  ├── feat/payment-module      ← Developer 2 (Phase 2b, parallel)
  └── feat/admin-module        ← Developer 3 (Phase 2b, parallel)
```

### Main Branch Protection Rules

- No direct push to main
- Merges only via PR
- PR must be reviewed by Tech Manager
- CI must pass
- No force push allowed

### Git Worktree for Parallel Development

Each developer uses Git Worktree to work in an isolated directory, avoiding file conflicts from branch switching:

```bash
# Project directory structure
~/projects/
├── my-project/                    # Main repo (main branch)
├── my-project-feat-user/          # Worktree (feat/user-module)
├── my-project-feat-payment/       # Worktree (feat/payment-module)
└── my-project-feat-admin/         # Worktree (feat/admin-module)
```

```bash
# Create worktree
git worktree add ../my-project-feat-user feat/user-module

# Clean up when done
git worktree remove ../my-project-feat-user
```

### Agent Auto Git Operations

Each GSD Agent in `/gsd auto` mode automatically:

```
1. Works on assigned branch
2. After each feature → git add + commit (clear commit message)
3. Periodically pushes to remote
4. All tasks done → creates PR to main
5. If Tech Manager comments with change requests → auto-fix and update PR
```

## 6. Dependency-Aware Task Splitting

### Splitting Principle

```
                    ┌──────────────┐
                    │  Phase 2a    │
                    │  Dependency  │    ← Must complete first
                    │  Layer       │
                    │  (1 developer)│
                    └──────┬───────┘
                           │ Merge to main
           ┌───────────────┼───────────────┐
           ▼               ▼               ▼
    ┌──────────┐    ┌──────────┐    ┌──────────┐
    │ Phase 2b │    │ Phase 2b │    │ Phase 2b │
    │ Module A │    │ Module B │    │ Module C │    ← All parallel
    │ Dev 1    │    │ Dev 2    │    │ Dev 3    │
    └──────────┘    └──────────┘    └──────────┘
```

### Splitting Rules

1. **Identify shared dependencies** — Code depended on by multiple modules (type defs, DB schema, base components, utilities) goes into the dependency layer
2. **Dependencies first** — Dependency layer completed by one developer and merged to main first
3. **Parallel tasks = developer count** — Split independent modules to exactly match the number of available developers
4. **Clear module boundaries** — Each module has its own directory, routes, APIs, no cross-module code edits
5. **Define interface contracts first** — Dependency layer defines inter-module interfaces (types, API schemas), each module implements per contract

### Splitting Examples by Project Type

**Web Frontend (React / Vue / Angular / Svelte):**
```
Dependency: Project init, routing config, shared components, API client, state skeleton
Parallel:   Split by page/feature module (users, orders, settings, reports...)
```

**Backend Services (Node.js / Python / Go / Rust / Java):**
```
Dependency: Project init, DB schema, middleware, shared utilities
Parallel:   Split by domain/service (auth, billing, notification, analytics...)
```

**Mobile Apps (React Native / Flutter / Swift / Kotlin):**
```
Dependency: Project init, navigation structure, design system, network layer
Parallel:   Split by feature module (home, profile, messaging, search...)
```

**Full-Stack Projects:**
```
Dependency: Monorepo init, shared types, database, API schema definitions
Parallel:   Frontend + Backend + Infrastructure in parallel (or vertical feature slices)
```

**CLI Tools / Libraries:**
```
Dependency: Project init, core interface definitions, config system
Parallel:   Split by subcommand/module
```

## 7. Fully Automated Execution Flow

### End-to-End Flow

```
1. Project Lead  → Provide PRD.md + developer count N
2. GSD Phase 0   → Tech selection + PREFERENCES.md + task assignment table
3. Project Lead  → Review and confirm
4. Developer 1   → /gsd auto → complete dependency layer → create PR
5. Tech Manager  → Review PR → merge to main
6. Developers 1~N → each: git worktree → /gsd auto → parallel dev → create PR
7. Tech Manager  → Review each PR → resolve conflicts → merge → verify acceptance
8. Project Lead  → Final confirmation → release
```

### Each Developer's Actions (Only 3 Steps)

```bash
# 1. Clone project
git clone <repo-url> && cd project

# 2. Create worktree (branch name from task assignment table)
git worktree add ../project-my-module feat/my-module

# 3. Start fully autonomous development
cd ../project-my-module && gsd auto
```

Everything after that is fully automated by the GSD Agent: code → test → commit → push → create PR.

## 8. Applicability

This best practice applies to all project types and programming languages:

| Type | Languages / Frameworks | Split Strategy |
|------|----------------------|----------------|
| Web Frontend | React, Vue, Angular, Svelte, Next.js, Nuxt | By page/feature module |
| Web Backend | Node.js, Python, Go, Rust, Java, C# | By domain/service |
| Full-Stack | Any frontend + backend combo | Vertical feature slices or layer separation |
| Mobile | React Native, Flutter, Swift, Kotlin | By feature module |
| Desktop | Electron, Tauri, Qt | By feature module |
| CLI / Library | Any language | By subcommand/module |
| Microservices | Any language | Natural service boundaries |
| Data Engineering | Python, Scala, SQL | By pipeline/ETL stage |
| Infrastructure | Terraform, Pulumi, Docker | By resource group/environment |

The core principle is the same: **Dependencies first → Maximize parallelism → Git collaboration → Tech Manager integration**.
