# GSD New Project Best Practices: From PRD to Team Delivery

A practical guide for starting new projects using multiple human developers + multiple GSD Agent teams working in collaboration.

---

## 1. GSD Configuration Hierarchy

GSD uses a **three-tier configuration system**. Understanding this is essential before starting any project.

### Tier 1: Global (`~/.gsd/`) — System-wide defaults

```
~/.gsd/
├── preferences.md              # Global preferences (mode, models, default skills)
├── web-preferences.json        # Web UI root directory lock
├── defaults.json               # System defaults
├── port-registry.json          # Global port assignment (>= 50000)
├── standards/                  # Engineering standards (STANDARDS.md, etc.)
│
├── agent/                      # Agent runtime configuration
│   ├── auth.json               # API keys (OpenRouter, GitHub, etc.)
│   ├── settings.json           # Provider, model, thinking level
│   ├── extensions/             # Agent extensions
│   └── skills/                 # Installed skills (symlinked from skillshare)
│
└── projects/                   # Per-project state (auto-managed by GSD)
    └── {HASH_ID}/
        ├── gsd.db              # SQLite database (project state)
        ├── PROJECT.md          # Project summary (auto-generated)
        ├── STATE.md            # Current state (auto-generated)
        ├── DECISIONS.md        # Technical decisions log
        ├── KNOWLEDGE.md        # Accumulated project knowledge
        ├── CONTEXT.md          # Auto-generated context
        ├── REQUIREMENTS.md     # Requirements tracking
        ├── preferences.md      # Project-level GSD preferences
        ├── metrics.json        # Project metrics & skill telemetry
        ├── milestones/         # Milestone directories
        │   └── M001/
        │       ├── CONTEXT.md  # Milestone context
        │       └── DECISIONS.md
        ├── activity/           # Session activity logs (JSONL)
        ├── journal/            # Daily journal
        └── runtime/            # Runtime state
```

### Tier 2: Workspace (`~/.gsd/` inside workspace) — Per-user overrides

When GSD Portal launches a workspace, the orchestrator creates:

```
/home/{username}/.gsd/
├── web-preferences.json        # Locks devRoot to this workspace
└── agent/
    ├── auth.json               # User's API keys (3-tier priority: user > admin > default)
    └── settings.json           # User's agent settings
```

### Tier 3: Project (project root) — Per-project behavior

```
project/
├── PRD.md                      # Product requirements document (your only input)
├── PREFERENCES.md              # Project-level GSD behavior (auto-generated by GSD)
├── CLAUDE.md                   # Project-level Claude instructions (optional)
└── src/                        # Source code
```

> **Note:** `MILESTONES.md`, `CONTEXT.md`, `DECISIONS.md`, `STATE.md` etc. are auto-generated
> by GSD into `~/.gsd/projects/{HASH}/` — they live in GSD's internal state, not in your project repo.
> GSD reads and updates them automatically across sessions.

### Configuration Priority

```
Project PREFERENCES.md  >  ~/.gsd/projects/{hash}/preferences.md  >  ~/.gsd/preferences.md
     (highest)                    (project state)                       (global default)
```

## 2. The Correct Startup Flow

### Core Principle: PRD is the only human input — everything else is auto-generated by GSD

```
Human prepares      GSD Phase 0            Human review         GSD Phase 1+
──────────────     ──────────────         ──────────────       ──────────────
PRD.md        →    Tech stack selection    Confirm/adjust   →   Implement by milestone
                   Generate PREFERENCES.md  Assign team tasks    Multi-agent parallel
                   Milestone breakdown
```

### Step 1: Human Preparation — Just provide PRD.md

Place `PRD.md` in an empty project directory. This is the **only file** you need to provide. Do not manually create `PREFERENCES.md`, `.gsd/`, or any configuration files.

### Step 2: Launch GSD — Phase 0 Automation

Open GSD and send this kickoff prompt:

```
Read PRD.md and execute Phase 0 initialization:

1. Analyze requirements, output tech stack recommendation (framework, database,
   deployment) with reasoning
2. Based on the tech stack, auto-generate PREFERENCES.md including:
   - Role definition
   - always_use_skills (must-use skills)
   - prefer_skills (preferred skills)
   - skill_rules (conditional trigger rules)
   - custom_instructions (project constraints)
3. Break requirements into 3-5 milestones (M1-M5), each independently verifiable
4. For each milestone: task list, estimated complexity, parallelizable tasks, dependencies

Constraints:
- All code must be real business logic — no mock/stub
- Follow engineering standards in ~/.gsd/standards/
```

GSD will automatically:
- Read PRD → Analyze requirements → Decide tech stack
- Select appropriate skills based on the stack (e.g., choosing React → adds `react-best-practices`)
- Generate `PREFERENCES.md` at the project root
- Create milestone structure in `~/.gsd/projects/{hash}/milestones/`

### Step 3: Human Review

After tech selection and milestone breakdown, **human reviews and confirms**:
- Is the tech stack appropriate?
- Does the skills configuration in PREFERENCES.md need adjustment?
- Is the milestone breakdown reasonable? Need to merge or split?
- Which milestones can run in parallel vs. which have dependencies?

Once approved, proceed to implementation.

## 3. Multi-Human + Multi-Agent Team Collaboration Model

Real projects are completed by **multiple human developers** and **multiple GSD Agent workspaces** working together.

### Role Distribution

```
┌──────────────────────────────────────────────────────┐
│                 Project Lead (Human)                  │
│  - Provides PRD.md                                    │
│  - Reviews tech selection & milestones                │
│  - Assigns tasks to team members                      │
│  - Code review & merge decisions                      │
└──────────────────────┬───────────────────────────────┘
                       │ Assign milestones / modules
          ┌────────────┼────────────┐
          ▼            ▼            ▼
   ┌──────────┐ ┌──────────┐ ┌──────────┐
   │ Dev A     │ │ Dev B     │ │ Dev C     │
   │ (Human)   │ │ (Human)   │ │ (Human)   │
   │           │ │           │ │           │
   │ GSD       │ │ GSD       │ │ GSD       │
   │ Agent A   │ │ Agent B   │ │ Agent C   │
   └──────────┘ └──────────┘ └──────────┘
       M1            M2           M3
    User module   Payment module  Admin panel
```

### Collaboration Principles

| Principle | Description |
|-----------|-------------|
| **Split by milestone / module** | Each developer + GSD Agent owns an independent milestone or feature module |
| **Share the same PREFERENCES.md** | All Agents use identical project config for consistent code style and standards |
| **Share the same PRD.md** | Every Agent reads the full requirements context to avoid implementation drift |
| **Git branch isolation** | Each developer works on their own branch, merges to main via PR |
| **Humans own integration** | Interface alignment, conflict resolution, and final merges are human decisions |

### Task Assignment Prompt Template

After the project lead confirms milestones, assign tasks to each developer:

**For Developer A (owns M1):**
```
Open the project in GSD and send:

Read PRD.md and PREFERENCES.md.
You are responsible for implementing M1 (User module).

M1 contains these tasks:
1. User registration / login API
2. User list page
3. User detail editing

Create branch feat/m1-user-module, implement task by task.
Commit after each task. Create PR when all tasks are done.

Constraints:
- Follow all rules in PREFERENCES.md
- Implement interfaces with other modules per PRD data structures
- Do not modify code belonging to other modules
```

### Key Shared Files for Parallel Development

When multiple Agents work in parallel, establish shared contracts upfront:

```
project/
├── PRD.md                # All Agents read this
├── PREFERENCES.md        # All Agents share this config
├── src/
│   ├── lib/types.ts      # Shared type definitions (generated in Phase 1 skeleton)
│   ├── lib/api-client.ts # Shared API client interfaces
│   └── ...
```

> **Tip:** Phase 1 skeleton should be completed by **a single Agent**, generating the project
> structure and shared type definitions before distributing to multiple Agents for parallel work.

## 4. Phased Execution Flow

```
Phase 0              Phase 1               Phase 2                Phase 3
Automated Planning   Skeleton (1 Agent)    Parallel Impl (N Agents)  Integration
────────────────    ────────────────      ────────────────         ────────────────
PRD → Tech selection  Project scaffolding   Each milestone parallel   PR Review
→ PREFERENCES.md     Routes / page layout  One module per dev        Integration tests
→ Milestone breakdown Shared type defs      Independent branches      Deploy config
                     Base components        Continuous commits        Merge & release
```

### Phase 0: Automated Planning (see Step 2)

GSD auto-completes tech selection, generates PREFERENCES.md, and breaks down milestones.

### Phase 1: Skeleton (Single Agent)

**Completed by one developer + one GSD Agent**, producing the shared foundation for all Agents:

```
Read PRD.md and PREFERENCES.md, implement Phase 1 skeleton:

1. Initialize project (per tech stack selection)
2. Set up directory structure
3. Configure lint, TypeScript, git hooks
4. Create shared type definitions (lib/types.ts)
5. Create base routes and page shells
6. Create shared component skeletons

Commit to main branch when done — this becomes the starting point for all developers.
```

### Phase 2: Parallel Implementation (Multiple Agents)

Once skeleton is ready, **multiple developers work simultaneously in their own GSD workspaces**:

- Each pulls latest from main
- Creates their own feature branch
- GSD Agent reads PRD.md + PREFERENCES.md, implements their assigned milestone
- Creates PR when done

### Phase 3: Integration

```
All milestone PRs have been submitted. Execute integration:

1. /review — Review each PR individually
2. Resolve conflicts between PRs
3. Merge to main
4. /test — Run full test suite
5. Verify against PRD.md acceptance criteria item by item
6. Configure deployment
```

## 5. GSD Session Management

### Resuming Context in New Sessions

GSD automatically maintains project state in `~/.gsd/projects/{hash}/` (STATE.md, DECISIONS.md, etc.),
so context is auto-restored in new sessions. But it helps to state current progress explicitly:

```
Read PRD.md and PREFERENCES.md.

Current progress: M2 complete, continuing M3.
Last completed: [specific task]. Start from [next task].
```

### Quality Gate at Milestone Completion

Each developer runs this in GSD when their milestone is done:

```
M[N] implementation complete. Run quality gates:

1. /review — audit all changes in this milestone
2. /test — generate tests for new code, run full suite
3. Verify against PRD.md acceptance criteria for M[N]
4. List any deviations or open items
```

### Cross-Agent Sync

When multiple Agents work in parallel, sync periodically to avoid conflicts:

```
Pull latest main into current branch, resolve any conflicts.
Check if shared files (lib/types.ts etc.) have been updated by other team members.
Ensure compatibility.
```

## 6. Key Principles

| Principle | Practice |
|-----------|----------|
| **PRD is the only human input** | PREFERENCES.md, milestones, tech selection — all auto-generated by GSD |
| **Plan before code** | Phase 0 must be completed and human-approved before implementation starts |
| **Skeleton first** | Phase 1 by a single Agent produces the shared foundation, then distribute |
| **Milestone-driven** | Progress M1 → M2 → M3, don't mix work across milestones |
| **Branch isolation** | Each developer/Agent works on an independent branch |
| **Humans own integration** | Interface alignment, conflict resolution, PR merges are human decisions |
| **Shared contracts** | All Agents share PRD.md + PREFERENCES.md for consistency |
| **Reuse Skills** | Leverage installed skills (52+), don't reinvent the wheel |
| **No mocks** | All code must be real business logic |
| **Small commits** | One feature per commit — easy to review and rollback |

## 7. Quick Reference: End-to-End Flow

```
1. Human: Place PRD.md → Launch GSD
2. GSD:   Read PRD → Tech selection → Generate PREFERENCES.md → Break down milestones
3. Human: Review tech stack + milestones → Assign team tasks
4. Agent (single):  Phase 1 skeleton → commit to main
5. Agent (multiple): Each pulls main → creates feature branch → implements milestone in parallel
6. Human: Review PRs → Resolve conflicts → Merge → Integration tests → Release
```
