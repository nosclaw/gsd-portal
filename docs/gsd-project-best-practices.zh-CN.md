# GSD 团队自动化开发最佳实践

基于 PRD 文档，使用多人类开发者 + 多 GSD AI Agent 团队实现全自动化并行开发。

适用于所有编程语言和项目类型。

---

## 核心理念

```
唯一的人工输入是 PRD.md 和开发者人数。
其余一切 — 技术选型、任务拆分、Git 配置、编码、提交、协作、合并 — 全部自动化。
```

---

## 角色定义

每个项目包含三种角色。人类开发者和 GSD AI Agent 是明确区分的两种身份。

### 角色总览

| 编号 | 角色 | 身份 | 职责 | 运行方式 |
|------|------|------|------|----------|
| — | **项目负责人** | 人类 | 提供 PRD.md、确认 Phase 0 输出、最终发布决策 | 手动 |
| — | **Tech Manager** | GSD AI Agent | PR 审查、冲突解决、合并到 main、验收审查、依赖协调 | `/gsd auto` |
| 1 | **人类开发者 A** | 人类 | 在电脑 A 上启动 `/gsd auto`，监督 Agent A 的工作 | 启动后观察 |
| 1 | **GSD Agent A** | AI Agent | 在 worktree 中自动完成模块 A 的全部编码、测试、提交、PR | `/gsd auto` |
| 2 | **人类开发者 B** | 人类 | 在电脑 B 上启动 `/gsd auto`，监督 Agent B 的工作 | 启动后观察 |
| 2 | **GSD Agent B** | AI Agent | 在 worktree 中自动完成模块 B 的全部编码、测试、提交、PR | `/gsd auto` |
| N | **...** | ... | ... | ... |

```
┌────────────────────────────────────────┐

│                                        │
│  职责：                                 │
│  • 克隆仓库                             │
│  • 启动 /gsd auto                      │
│  • 监督 Agent 工作质量                   │
│  • 必要时人工干预                        │
│                                        │
│  ┌──────────────────────────────────┐  │
│  │       GSD AI Agent A             │  │
│  │                                  │  │
│  │  职责：                           │  │
│  │  • 在 Git Worktree 中自动编码     │  │
│  │  • 运行测试                       │  │

```

每台电脑 = 1 个人类开发者 + 1 个 GSD AI Agent，组成一个**开发单元**。

### Tech Manager（技术经理）

独立的 GSD AI Agent，**不写业务代码**，只负责集成和协调。
职责由 [`tech-manager` skill](./skills/tech-manager/SKILL.md) 定义：
- 持续监控所有开放的 PR
- 逐个审查代码质量、安全性、规范符合度
- 解决合并冲突，保留双方意图
- 合并到受保护的 main 分支
- 对照 PRD.md 验收标准逐条检查
- 所有模块合并后输出验收报告

## Git Worktree 模式

所有开发单元**必须**使用 Git Worktree 模式工作。这在 `PREFERENCES.md` 中强制声明：

```markdown
## git_workflow
- mode: worktree
- branch_protection: main
- merge_method: pr_only
- auto_push: true
- auto_pr: true
```

### Worktree 模式的含义

GSD Agent 在 `/gsd auto` 启动时自动：
1. 从主仓库创建 Git Worktree（独立工作目录）
2. 在分配的 feature 分支上工作
3. 不影响主仓库的 main 分支和其他 worktree
4. 完成后自动创建 PR，不直接合并

### 为什么必须用 Worktree

- **零冲突**：每个 Agent 在物理隔离的目录中工作，不存在分支切换问题
- **真正并行**：N 个 Agent 可以同时在 N 个目录中编码，互不干扰
- **安全**：Agent 无法影响 main 分支，所有变更必须经过 PR

### Phase 0：自动规划

由 [`team-kickoff` skill](./skills/team-kickoff/SKILL.md) 自动完成。

**输入：** PRD.md + 人类开发者人数 N

**自动输出：**

1. 技术选型（优先选择最新最流行的技术栈）
2. `PREFERENCES.md`（含 git worktree 模式配置）
3. 角色分配表 + 里程碑/Slice 分配
4. Git 仓库初始化（main 保护 + feature 分支）

### Phase 2a：依赖层

人类开发者 A 在电脑 A 上运行 `/gsd auto`。
GSD Agent A 自动在 worktree 中完成所有共享基础代码，创建 PR。
Tech Manager 审查后合并到 main。

### Phase 2b：并行开发

依赖层合并后，所有人类开发者**同时**在各自电脑上运行 `/gsd auto`：
- 每个 GSD Agent 自动创建 worktree，在独立目录中工作
- 每个 Agent 只在自己分配的分支和模块内编码
- 完成后自动创建 PR

### Phase 3：集成验收

Tech Manager 持续运行，自动完成：
- 逐个审查 PR → 合并 → 解决冲突
- 所有模块合并后运行完整测试
- 对照 PRD.md 验收标准出具验收报告

## 依赖感知拆分规则

```
                    ┌──────────────────────┐
                    │  M0 依赖层            │    ← 必须先完成
                    │  开发单元 1            │
                    │  (人类 A + Agent A)   │
                    └──────────┬───────────┘
                               │ PR → Tech Manager 合并到 main
               ┌───────────────┼───────────────┐
               ▼               ▼               ▼
     ┌─────────────┐  ┌─────────────┐  ┌─────────────┐
     │ M1 用户模块  │  │ M2 支付模块  │  │ M3 管理后台  │  ← 全部并行
     │ 开发单元 1   │  │ 开发单元 2   │  │ 开发单元 3   │
     │ 人类A+AgentA│  │ 人类B+AgentB│  │ 人类C+AgentC│
     └─────────────┘  └─────────────┘  └─────────────┘
```

**拆分规则：**
- 被多个模块依赖的代码归入依赖层（类型、schema、基础组件、中间件）
- 并行模块数 = 人类开发者人数 N，最大化并行度
- 每个模块对应一个里程碑（M1, M2, M3...），每个里程碑拆分为多个 Slice（S1.1, S1.2...）
- 每个模块有独立的目录、路由、API，不允许跨模块编辑
- 依赖层定义模块间的接口契约，各模块按契约实现

## 技术选型原则

Phase 0 技术选型时，遵循以下优先级：

| 优先级 | 原则 | 示例 |
|--------|------|------|
| 1 | **优先最新、最流行的技术** | bun > pnpm > yarn > npm |
| 2 | **选择性能更好的方案** | Bun runtime > Node.js（如果兼容） |
| 3 | **选择开发体验更好的框架** | Next.js > CRA, Nuxt > Vue CLI |
| 4 | **选择类型安全的方案** | TypeScript > JavaScript, Rust > C |
| 5 | **PRD 有明确指定时，遵循 PRD** | PRD 要求 Python → 用 Python |

## GSD 配置层级

```
项目 PREFERENCES.md  >  ~/.gsd/projects/{hash}/preferences.md  >  ~/.gsd/preferences.md
    （最高优先级）              （项目状态，GSD 自动管理）            （全局默认）
```

| 层级 | 路径 | 管理方式 |
|------|------|----------|
| 全局 | `~/.gsd/` | 系统默认值、API 密钥、已安装 skills |
| 项目状态 | `~/.gsd/projects/{hash}/` | GSD 自动管理（里程碑、状态、决策、知识） |
| 项目配置 | 项目根目录 `PREFERENCES.md` | Phase 0 自动生成 |

## 适用范围

| 项目类型 | 依赖层 | 并行层拆分方式 |
|----------|--------|---------------|
| Web 前端 | 路由、共享组件、API client、状态骨架 | 按页面/功能模块 |
| Web 后端 | DB schema、中间件、公共工具 | 按 domain/service |
| 全栈 | monorepo 初始化、共享类型、API schema | 前后端分离或功能垂直切分 |
| 移动端 | 导航、设计系统、网络层 | 按功能模块 |
| 桌面端 | 窗口框架、IPC、共享状态 | 按功能模块 |
| CLI / 库 | 核心接口、配置系统 | 按子命令/模块 |
| 微服务 | proto/schema 定义、CI 模板 | 天然按服务拆分 |
| 数据工程 | pipeline 框架、共享连接器 | 按 pipeline 阶段 |
| 基础设施 | provider 配置、共享 module | 按资源组/环境 |

## 速查

```
1. 项目负责人 → 提供 PRD.md + 人类开发者人数 N
2. /team-kickoff → 自动生成一切（选型、PREFERENCES.md、分工表、Git）
3. 项目负责人 → 确认分工表
4. 电脑 A：人类开发者 A → /gsd auto → Agent A 完成依赖层 → PR
5. Tech Manager → /gsd auto → 审查 → 合并依赖层到 main
6. 电脑 A~N：所有人类开发者 → 各自 /gsd auto → 各 Agent 并行开发 → 各自 PR
7. Tech Manager → 逐个审查 → 冲突解决 → 合并 → 验收报告
8. 项目负责人 → 确认验收 → 发布
```
