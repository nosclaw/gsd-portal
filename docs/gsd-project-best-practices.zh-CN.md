# GSD 团队自动化开发最佳实践

基于 PRD 文档，使用多人类开发者 + 多 GSD AI Agent 团队实现全自动化并行开发。

适用于所有编程语言和项目类型。

---

## 核心理念

```
唯一的人工输入是 PRD.md 和开发者人数。
其余一切 — 技术选型、任务拆分、Git 配置、编码、提交、协作、合并 — 全部自动化。
```

---

## 角色定义

每个项目由以下角色组成：

| 角色 | 类型 | 职责 | 运行方式 |
|------|------|------|----------|
| **项目负责人** | 人类 | 提供 PRD.md、确认 Phase 0 输出、最终发布决策 | 手动 |
| **Tech Manager** | GSD AI Agent | PR 审查、冲突解决、合并到 main、验收审查、依赖协调 | `/gsd auto` |
| **开发者 1~N** | 人类 + GSD AI Agent | 各自负责一个独立模块的全部实现 | `/gsd auto` |

### Tech Manager

专职集成角色，**不写业务代码**。职责由 [`tech-manager` skill](./skills/tech-manager/SKILL.md) 定义：
- 持续监控所有开放的 PR
- 逐个审查代码质量、安全性、规范符合度
- 解决合并冲突，保留双方意图
- 合并到受保护的 main 分支
- 对照 PRD.md 验收标准逐条检查
- 所有模块合并后输出验收报告

### 开发者

每个开发者在自己的电脑上运行 `/gsd auto`，GSD Agent 全自动完成：
- 在分配的分支上编码
- 每个功能点完成后 commit + push
- 所有任务完成后创建 PR 到 main
- 收到 Tech Manager 的修改意见后自动修复

## 自动化流程

```
                        PRD.md + 开发者人数 N
                                │
                    ┌───────────▼───────────┐
                    │   /team-kickoff       │  ← GSD Skill 自动执行
                    │                       │
                    │  • 技术选型            │
                    │  • 生成 PREFERENCES.md │
                    │  • 依赖感知任务拆分    │
                    │  • Git 分支 + 保护     │
                    │  • 角色分配表          │
                    └───────────┬───────────┘
                                │
                    ┌───────────▼───────────┐
                    │   项目负责人审核确认     │  ← 唯一的人工检查点
                    └───────────┬───────────┘
                                │
              ┌─────────────────┼─────────────────┐
              ▼                                   ▼
    ┌──────────────────┐                ┌──────────────────┐
    │  开发者 1         │                │  Tech Manager    │
    │  /gsd auto       │                │  /gsd auto       │
    │                  │                │                  │
    │  Phase 2a:       │                │  持续运行：        │
    │  依赖层 → PR     │───── PR ──────▶│  审查 → 合并      │
    └────────┬─────────┘                │                  │
             │ 依赖层合并后              │                  │
    ┌────────┼────────┐                 │                  │
    ▼        ▼        ▼                 │                  │
  Dev 1    Dev 2    Dev N               │                  │
  /gsd     /gsd     /gsd               │                  │
  auto     auto     auto               │                  │
    │        │        │                 │                  │
    └── PR ──┴── PR ──┴───── PR ──────▶│  审查 → 合并      │
                                        │  冲突解决          │
                                        │  验收报告          │
                                        └──────────────────┘
```

### Phase 0：自动规划

由 [`team-kickoff` skill](./skills/team-kickoff/SKILL.md) 自动完成，输入只需 PRD.md 和开发者人数：

1. 分析 PRD → 技术选型（语言、框架、数据库、部署）
2. 基于选型自动选择合适的 skills → 生成 `PREFERENCES.md`
3. 依赖感知拆分：
   - 共享依赖（类型、schema、基础组件）→ **依赖层**（1 个开发者先完成）
   - 独立模块拆为 N 份 → **并行层**（N 个开发者同时开发）
4. 配置 Git：main 分支保护 + 创建 feature 分支
5. 输出角色分配表

### Phase 2a：依赖层

开发者 1 运行 `/gsd auto`，完成所有共享基础代码，创建 PR。
Tech Manager 审查后合并到 main。

### Phase 2b：并行开发

所有开发者**同时**运行 `/gsd auto`：
- GSD 自动使用 Git Worktree 在独立目录中工作
- 每个 Agent 只在自己的分支和模块内编码
- 完成后自动创建 PR

### Phase 3：集成验收

Tech Manager 持续运行，自动完成：
- 逐个审查 PR → 合并 → 解决冲突
- 所有模块合并后运行完整测试
- 对照 PRD.md 验收标准出具验收报告

## 依赖感知拆分

```
                    ┌──────────────┐
                    │  依赖层       │    ← 必须先完成
                    │  (开发者 1)   │
                    └──────┬───────┘
                           │ 合并到 main
           ┌───────────────┼───────────────┐
           ▼               ▼               ▼
    ┌──────────┐    ┌──────────┐    ┌──────────┐
    │ 模块 A   │    │ 模块 B   │    │ 模块 C   │    ← 全部并行
    │ 开发者 1  │    │ 开发者 2  │    │ 开发者 3  │
    └──────────┘    └──────────┘    └──────────┘
```

**拆分规则：**
- 被多个模块依赖的代码归入依赖层（类型、schema、基础组件、中间件）
- 并行任务数 = 开发者人数，最大化并行度
- 每个模块有独立的目录、路由、API，不允许跨模块编辑
- 依赖层定义模块间的接口契约，各模块按契约实现

## GSD 配置层级

```
项目 PREFERENCES.md  >  ~/.gsd/projects/{hash}/preferences.md  >  ~/.gsd/preferences.md
    （最高优先级）              （项目状态，GSD 自动管理）            （全局默认）
```

| 层级 | 路径 | 管理方式 |
|------|------|----------|
| 全局 | `~/.gsd/` | 系统默认值、API 密钥、已安装 skills |
| 项目状态 | `~/.gsd/projects/{hash}/` | GSD 自动管理（里程碑、状态、决策、知识） |
| 项目配置 | 项目根目录 `PREFERENCES.md` | Phase 0 自动生成 |

---

## 适用范围

| 项目类型 | 依赖层 | 并行层拆分方式 |
|----------|--------|---------------|
| Web 前端 | 路由、共享组件、API client、状态骨架 | 按页面/功能模块 |
| Web 后端 | DB schema、中间件、公共工具 | 按 domain/service |
| 全栈 | monorepo 初始化、共享类型、API schema | 前后端分离或功能垂直切分 |
| 移动端 | 导航、设计系统、网络层 | 按功能模块 |
| 桌面端 | 窗口框架、IPC、共享状态 | 按功能模块 |
| CLI / 库 | 核心接口、配置系统 | 按子命令/模块 |
| 微服务 | proto/schema 定义、CI 模板 | 天然按服务拆分 |
| 数据工程 | pipeline 框架、共享连接器 | 按 pipeline 阶段 |
| 基础设施 | provider 配置、共享 module | 按资源组/环境 |

## 速查

```
1. 项目负责人 → 提供 PRD.md + 开发者人数
2. /team-kickoff → 自动生成一切（选型、PREFERENCES.md、分工、Git）
3. 项目负责人 → 确认
4. 开发者 1 → /gsd auto → 依赖层 → PR
5. Tech Manager → /gsd auto → 审查 → 合并
6. 所有开发者 → /gsd auto → 并行开发 → PR
7. Tech Manager → 审查 → 冲突解决 → 合并 → 验收报告
8. 项目负责人 → 确认 → 发布
```
